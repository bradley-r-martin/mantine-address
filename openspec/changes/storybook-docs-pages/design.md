## Context

The library uses Storybook as both a development harness and the primary documentation surface for `AddressInput` and provider integrations. Existing stories already cover key concepts (manual entry, restrictions, prefill, providers), but the presentation is story/demos-first rather than docs-page-first.

The goal of this change is to make Storybook read like a component documentation site: structured pages with clear narrative, props guidance, and copy/paste-ready snippets, while keeping the same interactive demos and prop showcases that already exist.

Constraints:

- This is a reusable Mantine-first component library; story presentation should follow Mantine patterns and remain unopinionated beyond documentation layout.
- Prefer additive Storybook configuration changes (avoid forcing consumers or large rework).
- Keep provider setup complexity in provider-specific pages.

## Goals / Non-Goals

**Goals:**

- Convert the `AddressInput` story set into documentation-quality “pages” with consistent layout (overview, usage, props, examples).
- Keep existing demo components and examples, but present them within a standardized doc layout.
- Add explicit code snippets for each example that reflect recommended usage patterns (imports, minimal props, and the feature being documented).
- Improve information scent and navigation: predictable page structure, sectioning, and story grouping.

**Non-Goals:**

- Change `AddressInput`’s runtime behavior, props, or exported API.
- Rebuild the story examples into a new demo app; keep stories concise and focused.
- Add new providers or significantly alter provider implementation beyond documentation/presentation.

## Decisions

### Decision: Use Storybook Docs pages (MDX) for narrative + keep CSF stories for interactivity

**Choice:** Introduce documentation pages using Storybook Docs (MDX) as the primary narrative surface, and reference existing CSF stories/examples within those pages.

**Rationale:**

- MDX supports rich documentation structure (headings, tables, sections) in a way that CSF-only cannot.
- Existing CSF stories remain interactive and args-driven, preserving the current demo behavior and enabling controls.
- This keeps “demo code” reusable and avoids duplicating logic in MDX.

**Alternatives considered:**

- CSF-only “docs pages” using long `render` stories: rejected due to poor narrative structure and higher maintenance.
- Autogenerated docs only (argTypes tables): rejected because it doesn’t provide guidance or curated example snippets.

### Decision: Standardize a shared docs layout component for consistent look & feel

**Choice:** Create a small set of Storybook-only layout primitives (e.g. `DocsPage`, `DocsSection`, `ExampleBlock`) implemented with Mantine components, used by all MDX docs pages.

**Rationale:**

- Consistency across docs pages without copy/paste.
- Enables “beautiful pages” with a coherent visual system (spacing, typography, callouts) aligned with Mantine.
- Keeps styling within Storybook scope (no impact on library runtime bundle).

**Alternatives considered:**

- Pure markdown/MDX with minimal styling: rejected (doesn’t meet “beautiful pages” requirement).
- Heavy custom CSS/theme override: rejected (harder to maintain, less composable than Mantine primitives).

### Decision: Code snippets are curated and authored (not inferred) and live next to examples

**Choice:** Add explicit snippet strings (or lightweight snippet components) alongside each example, rendered in the docs page next to the live demo.

**Rationale:**

- Ensures snippet correctness and readability (imports, props, constants usage).
- Avoids trying to reliably extract code from React components (fragile, often unreadable).
- Keeps snippets aligned with “recommended pattern” rather than “whatever the demo needs”.

**Alternatives considered:**

- Automatic snippet extraction (source loader): rejected due to brittleness and noisy output.
- Embedding large blocks of code directly in MDX: allowed for small snippets but rejected as the primary approach (duplication risk).

### Decision: Preserve existing story group taxonomy, but add “Docs” entry points

**Choice:** Keep current groupings like `AddressInput/*` and `Providers/*`, but add top-level docs pages (or `*.mdx`) that become the “start here” entries for each group (e.g. `AddressInput/Docs`).

**Rationale:**

- Minimal disruption to existing navigation while improving the first-click experience.
- Maintains compatibility with current story references and developer muscle memory.

## Risks / Trade-offs

- **Docs addon / MDX compatibility risk** → Mitigation: use the current Storybook Docs addon approach supported by the repo’s Storybook version; keep changes incremental and validate build output.
- **Snippet drift from actual behavior** → Mitigation: keep snippets small and pattern-focused; colocate snippet definitions with the example story file; prefer using shared constants/imports that match runtime usage.
- **Overengineering story layout** → Mitigation: limit shared docs primitives to a small set; keep story logic in existing story/demo files.
- **Maintenance overhead** → Mitigation: reuse existing demo components and do not duplicate full examples in MDX; only curate key snippets.
